# 前后端 / 客户端 / 数据库的关系

## 餐厅类比

```
顾客（用户）→ 服务员（前端/客户端）→ 厨房（后端）→ 冰箱/仓库（数据库）
```

---

## 各层职责

### 前端/客户端 = 服务员 + 菜单 + 餐厅装修

用户能看到、能摸到的一切。网页、App 界面、按钮、动画。它的职责：
- 把信息漂亮地展示给用户
- 收集用户的操作（点了什么、输了什么）
- 把请求传给后端

### 后端 = 厨房

用户看不到，但干真正的活。它的职责：
- 处理业务逻辑（下单、扣款、验证身份）
- 决定该给前端返回什么数据
- 跟数据库打交道（存取数据）
- 调用外部服务（支付、短信）

### 数据库 = 仓库/冰箱

只负责**存东西和取东西**。用户信息、订单记录、文章内容，全在这里。它不思考，只听后端的指令：存这条、取那条、删掉这个。

---

## 配合流程（以「用户登录」为例）

```
1. 用户在前端输入账号密码，点登录
2. 前端把账号密码发给后端
3. 后端拿着账号去数据库查：这人存在吗？密码对吗？
4. 数据库返回查询结果给后端
5. 后端判断结果，告诉前端：「成功」或「密码错误」
6. 前端根据结果显示：跳转首页 或 弹出错误提示
```

---

## 前端 vs 客户端

基本是一回事，只是场景不同：
- 网页端叫**前端**（浏览器里跑的）
- 手机端叫**客户端**（App）
- 本质都是「用户直接交互的那一层」

---

## 一句话总结

**前端负责「好看」，后端负责「好用」，数据库负责「记住」。**

---

## Mock 数据与联调

### Mock 数据

前端开发时用的模拟数据，模拟后端接口返回，让前端不用等后端写完就能把页面逻辑跑通。

Mock 的本质是：**在真实接口还没好的时候，让前端不被阻塞，实现并行开发。**

后端也会 mock 外部依赖（如第三方支付接口、短信服务）来测试自己的逻辑。

**谁依赖了不受自己控制的东西，谁就需要 mock。**

### 联调

前后端真实对接，用真实接口、真实数据库。验证两边对接口的理解是否一致：
- 字段名对不对
- 数据格式对不对
- 错误码处理对不对
- 鉴权流程通不通

### 开发流程

1. 先约定好接口文档（字段、格式、状态码）
2. 前端按文档写 mock，自己跑通页面逻辑
3. 后端按文档实现真实接口
4. 两边都写完了，联调 — 验证双方对文档的理解一致

**每完成一个模块就联调一次，不要攒到最后。** 问题当场暴露当场修。
